  0. EXECUTIVE SUMMARY  —  Three Key Findings
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  This work ports the AmorE protocol from BN254 to BLS12-381 on the
  STM32F407 Cortex-M4 platform. Three findings stand out:


  ▌ FINDING 1 — Protocol correctness
  
    Validation results from a 92-minute benchmark with 61 honest rounds
    plus 1 malicious round, on the binary at SHA256 7303fa43...
    
        • 61 / 61  honest rounds verified            (100.0%)
        •  1 /  1  malicious rounds rejected         (100.0%)
        •  0       errors of any kind (CRC, UART, timeout, bad CMD)
        •  8 /  8  implementation validation suite checks pass
        •  457 /457 host-side unit tests pass
    
    The validation suite cross-checks 20 curve constants against
    py_ecc.bls12_381 (independent reference) and verifies that
    malicious-round rejection occurs within the Verify computation
    (last_error=0, sec_verify_result=0), not via crash or timeout.
    
    The BN254 -> BLS12-381 port required only field/curve/scalar
    arithmetic changes; the protocol logic in amore.c is unchanged.


  ▌ FINDING 2 — Memory footprint
  
    The complete BLS12-381 implementation occupies:
    
        Code (.text)        :  15.5 KiB   (1.48% of 1024 KiB Flash)
        Initialized data    :   0.02 KiB
        Zero-init data (.bss):  35.0 KiB   (17.85% of 192 KiB SRAM)
        Maximum stack frame :    388 B
        ───────────────────────────────────────────
        Total Flash         :  ~15.5 KiB    (98.5% headroom)
        Total SRAM          :  ~35.5 KiB    (82.0% headroom)
    
    Memory remaining for application code: 1008 KiB Flash,
    156 KiB SRAM (out of 1024 KiB and 192 KiB respectively).


  ▌ FINDING 3 — Slowdown vs BN254 baseline
  
    Comparing to the BN254 baseline on identical hardware, the 254→381
    bit field expansion produces a 5.03× slowdown in per-round amortized
    timing. This matches the theoretical prediction:
    
        Field expansion:    254 → 381 bits     (1.50× more bits)
        Limb count:           8 → 12 limbs     (1.50×)
        Multiplication O(n²): 12² / 8² = 2.25× (CIOS Montgomery)
        Assembly removal:                        ~2× (BN254 had asm)
        ─────────────────────────────────────────────────────
        Predicted slowdown:                     ~4.5–5.5×
        Measured slowdown:                       5.03×  ✓
    
    A measured ratio outside the predicted 4.5-5.5x range would
    warrant investigation: too-slow could indicate a bug; too-fast
    an unaccounted optimization. The agreement between prediction
    and measurement is consistent with the inner-loop arithmetic
    being correctly implemented, complementary to the unit tests
    in Section 3.

  3 x direct pairing:                                  1,570.2 ms
  AmorE per-round amortized (this work, N=50):         1,919.3 ms

  AmorE / 3x direct ratio: 1.22x.

  On Cortex-M4 with BLS12-381, AmorE per-round exceeds the cost of
  3 direct pairings. The client-side work consists of scalar
  multiplications and one Fp12 exponentiation; pairings are
  delegated to the server. Whether delegation is preferable in
  this regime depends on factors not measured here (energy,
  server trust assumptions, network constraints, latency).

  Operation               BN254          BLS12-381    Slowdown
  ─────────────────────  ──────────    ────────────  ──────────
  OneTimeSetup            503.9 ms      2,565.2 ms    5.09×
  Blind (per round)       197.3 ms      1,032.1 ms    5.23×
  Verify (per round)      175.2 ms        887.2 ms    5.06×
  Per-round amortized     381.8 ms      1,919.3 ms    5.03×

  Sources of slowdown (theoretical model):
    - Field size:     254 → 381 bits  ( ~50% larger)
    - Limb count:       8 → 12        (  50% more)
    - fp_mul cost:    O(n²) → 12²/8²  = 2.25× per multiplication
    - Assembly:       hand-tuned → none (BN254 had asm; BLS12-381 is C)
    - Buffer sizes:   roughly 2× throughout
    
    Compounded: ~4.5×–5.5× expected slowdown range.
    Measured:    5.03×  →  WITHIN RANGE.

  Why this matters:
    A measured ratio outside the 4.5–5.5× window would warrant
    investigation: either a port bug (if too slow) or an unaccounted
    optimization (if too fast). The 5.03× match between prediction and
    measurement provides indirect confirmation that the inner loops of
    fp_mul, fp_sqr, fp2_mul, and fp12_mul are correctly implemented —
    independent of and complementary to the unit tests in Section 3.

  8.1 The flag that makes the difference: -O2 vs -O3
  ──────────────────────────────────────────────────

  The root cause of the 2.14x performance difference is the GCC
  optimization level. With -O3, GCC unrolls the inner loops of the
  CIOS Montgomery multiplication in fp_mul. With -O2, those inner
  loops are kept as runtime loops. Same source code, same correctness,
  different code-size/speed trade-off.

    Variant A  (CMAKE_BUILD_TYPE unset, -O2):
      Optimization: -O2 (GCC moderate)
      fp_mul size:  376 bytes (looped)
      Amort/round:  1,919.3 ms
      Tag:          measurement-O2-2026-05-12
      Binary SHA:   6ce9f8decf274e3286eedc59bab695ccb7c3cf10e1d4817097011ab1a23a62ff
      Build:        cmake -B build/O2 \
                          -DCMAKE_TOOLCHAIN_FILE=cmake/toolchain-stm32f4.cmake \
                          -DCURVE=BLS12_381

  8.3 Why -O3 produces a 2.14x speedup
  ────────────────────────────────────

  Forensic disassembly of fp_mul from both binaries:

    Variant A (-O2):
      131 instruction lines
      2 UMLAL instructions in code (inside loops)
      8 backward branches (CIOS inner loops compiled as runtime loops)
      Layout: looped — outer loop iterates 12 times,
              inner mul/reduce loops also iterate 12 times each

    Variant B (-O3):
      469 instruction lines
      24 UMLAL instructions in code (12x more, inner loops unrolled)
      14 backward branches (only outer loop remains; 12 of the 14
        branches are co-located early-exit checks)
      Layout: outer loop iterates 12 times,
              inner mul/reduce loops fully unrolled inline

  Both variants execute approximately the same total work per fp_mul
  call: ~288 UMLAL operations (12 outer iterations x 12 inner
  iterations x 2 inner loop bodies). The speedup comes from
  eliminating per-iteration overhead in the inner loops:

    - Loop counter increment + compare + branch (3 instructions/iter)
    - Pipeline stalls on conditional branches
    - Register pressure from maintaining loop variables

  GCC's -O3 enables the loop-unrolling optimization (-funroll-loops
  implicitly, plus inlining heuristics that favor expanding small
  hot loops). -O2 keeps the loops as loops, prioritizing code size
  over the inner-loop speedup.


  8.4 Note on terminology in commit history
  ─────────────────────────────────────────

  Commit ffb7a0f's message refers to "VFP register optimization
  restored." Disassembly inspection shows this is a misnomer; both
  variants use UMLAL (integer multiply-accumulate) for the actual
  multiplication. The Cortex-M4 FPU (FPv4-SP-D16) is single-precision
  float only and is not applicable to 32x32 -> 64 unsigned integer
  arithmetic. The d8 VFP register usage observed in Variant A is
  limited to a single PC-relative constant load and has no impact
  on multiplication performance.

  The real distinction documented in this section is -O2 vs -O3.


  8.5 Effect on Section 5 (vs RELIC direct pairing)
  ─────────────────────────────────────────────────

  Variant A (-O2):  AmorE 1,919 ms / 3xdirect 1,570 ms = 1.22x (slower)
  Variant B (-O3):  AmorE   898 ms / 3xdirect 1,570 ms = 0.572x (faster)

  The 1,570 ms RELIC baseline was measured against a pre-2026-05-12
  RELIC build, likely also at -O2. RELIC's internal Montgomery
  multiplication uses similar 12-limb arithmetic and would likely
  show a comparable -O2 -> -O3 speedup. Re-measurement of RELIC
  with -O3 is planned for the energy-study phase.


  8.6 Effect on Section 6 (vs BN254)
  ──────────────────────────────────

  Variant A (-O2):  BLS 1,919 ms / BN254 381.8 ms = 5.03x slowdown
  Variant B (-O3):  BLS   898 ms / BN254 381.8 ms = 2.35x slowdown

  The BN254 baseline (381.8 ms) was measured on 2026-04-01, also
  pre-2026-05-12 (likely -O2). A direct apples-to-apples comparison
  requires re-measuring BN254 with -O3. This is currently blocked
  because fp12_mul on the unified branch is BLS12-381-specific
  (see docs/future_work.md).
