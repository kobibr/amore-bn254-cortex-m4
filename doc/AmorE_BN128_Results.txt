  Note added 2026-05-13:

  AmorE BN254 below uses GCC -O2 throughout (Section 1). 

  The RELIC BN254 baseline cited in Section 11 (252.3 ms) is built with 
  the arm-asm-254 backend — hand-tuned ARM assembly for field arithmetic
  (fp_mul_low, fp_add_low, fp_rdc_low as .s files). The -O2 / -O3 GCC
  flag affects only the surrounding C code, not the assembly. The 252.3 ms
  reflects RELIC's hand-tuned ASM performance, which is RELIC's intended
  production configuration.

  The Section 11 comparison is therefore "AmorE C at -O2" vs "RELIC C-with-
  hand-tuned-ASM-backend". It is legitimate (both are production builds)
  but is implementation-asymmetric (C vs ASM).

  This is different from the BLS12-381 situation documented in
  AmorE_BLS12_381_Results.txt Section 8.5, where RELIC uses ARITH=easy
  (pure C at -O3) and AmorE was re-built at -O3 for clean -O3 vs -O3
  comparison.

  Future work (docs/future_work.md): hand-tuned ASM optimization of
  AmorE's fp_mul (projected ~30% additional speedup, target ~268 ms
  per round for BN254 if scaled from BLS).

  11.2  Why AmorE still has value

  AmorE is not faster than local C — but speed was never the only
  axis. The protocol's value lies elsewhere:

  Resource              AmorE client    RELIC pairing    Ratio
  --------------------- --------------- ---------------- -------------
  Flash (.text)         18.5 KB         55.1 KB          AmorE: 3.0x lighter
  SRAM (.bss)           3.1 KB          101.4 KB         AmorE: 33x lighter
  ELF total             21.7 KB         156.7 KB         AmorE: 7.2x smaller

  In addition, AmorE provides:
    - Input privacy: server never sees plaintext (A,B)
    - Cheating detection: malicious results rejected with prob ~ 1
    - Feasibility on parts where RELIC does not fit (e.g. STM32L0/G0
      with 4-32 KB SRAM)

  11.3  Honest summary

  The right framing for AmorE on STM32F4 is therefore:

      "Trade ~1.5x speed (vs in-place C) for 33x less RAM,
       3x less Flash, input privacy, and the ability to outsource
       pairing computation to an untrusted helper."

  On a Cortex-M4 with 192 KB SRAM, this trade-off is not compelling
  for speed-sensitive applications. It becomes compelling on:
    - smaller MCUs where 100 KB of pairing-state cannot fit
    - settings where input privacy is required
    - settings where battery cost of computation matters more than
      latency

  Two distinct comparisons, each measured on real hardware:

  vs Python server on RPi  (Section 7)
    Speedup            :  192.4x   (systemic, deployment-specific)
    AmorE cost ratio   :  0.52%    of remote pairing time

  vs Optimized C on same STM32  (Section 11)
    Speedup            :  0.66x    (i.e. AmorE is 1.51x SLOWER)
    Memory ratio       :  AmorE uses 33x less RAM and 3x less Flash

  The contribution of AmorE is therefore best characterized as
  ENABLING pairing-based cryptography on memory-constrained MCUs
  where a full pairing library does not fit, AND on supplying input
  privacy to the outsourced computation.  It is not a speed
  optimization for parts where RELIC already fits comfortably.
