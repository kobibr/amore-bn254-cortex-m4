  intended use, the single case (T.3) is the worst case. Both are correct.

    1) vs the PYTHON server (Raspberry Pi)

    2) vs OPTIMIZED LOCAL C, ONE pairing, our hand-written Fp12
         AmorE is 1.75x SLOWER              (Section 11)
    3) vs LOCAL C, a BATCH of 50, RELIC-grade arithmetic
         AmorE SAVES 37% of compute time    (Section 12 / T.1)

  AmorE beats a slow (Python) server easily; loses to fast local C for a
  SINGLE pairing with today's code; wins again once you delegate MANY
  pairings at once (batch). All three are true.

  Note revised 2026-05-31:

  AmorE BN254 below uses GCC -O2 throughout (Section 1).

  The RELIC BN254 baseline in Section 11 is built with ARITH=easy
  (pure C) — the same backend used for the BLS12-381 comparison.

  An earlier revision of this note (2026-05-13) attributed the baseline
  to a hand-tuned arm-asm-254 assembly backend. That attribution was
  incorrect: RELIC's arm-asm-254 sources do not assemble for the
  Thumb-2 / Cortex-M4 target (legacy conditional-infix syntax, e.g.
  `adcnes`), so no assembly backend was ever built or measured on this
  platform. Every RELIC build used here is ARITH=easy (pure C).

  The figure originally reported in Section 11 (252.3 ms / 42,386,215
  cyc) reflected an earlier easy-C build. Re-measurement on 2026-05-31
  with the current easy-C RELIC (ARITH=easy, FP_PRIME=254) gives
  218.2 ms (36,658,970 cyc) at -O2; -O3 is within 0.3% (36,778,389 cyc).
  Section 11 has been updated to 218.2 ms.

  The Section 11 comparison is therefore C-vs-C (symmetric): AmorE pure
  C at -O2 vs RELIC pure C (easy) — not C-vs-ASM.

  IMPORTANT: every AmorE timing in this report (Sections T.2, T.3, 4, 11)
  is from the GCC -O2 AmorE build (see Section 1, line "Flags: -O2 ...").
  The 2026-06-04 Energy Study measured AmorE at -O3 on the same STM32F4.
  Those -O3 numbers are reported separately in docs/FINAL_RESULTS and the
  repo README; they are NOT interchangeable with the -O2 figures below.

  Side-by-side (BN254, both measured by the same DWT counter @ 168 MHz):

    Quantity                       -O2 (this report)   -O3 (Energy Study)
    ----------------------------   -----------------   ------------------
    RELIC pp_map_oatep_k12 (1×)    218.2 ms            218.92 ms
                                   (36,658,970 cyc)    (36,778,389 cyc)
    AmorE amort/round (N=50)       381.8 ms            422.11 ms
                                   (64,147,052 cyc)    (70,915,190 cyc)
    AmorE / RELIC ratio            1.75×               1.93×

  The RELIC pairing matches across builds (easy-C is O-level insensitive,
  within 0.3%). The AmorE per-round figure does NOT: at -O3 it is ~10.6%
  SLOWER than at -O2 (422.11 vs 381.8 ms).

  >>> Flagged anomaly (not yet root-caused) <<<
  This is the OPPOSITE of BLS12-381, where -O3 was ~2.14× FASTER than -O2
  (1,919 → 898 ms, via fp_mul inner-loop unrolling). For BN254 (8 limbs),
  -O3 appears to regress the per-round compute. A plausible cause is that
  the unrolling/inlining that helps BLS's 12-limb fp_mul does not help —
  or hurts — the 8-limb case (e.g. I-cache pressure or register spilling),
  but this has NOT been confirmed by disassembly here. Until a fp_mul
  -O2-vs-O3 disassembly comparison on BN254 is done, treat the -O3 BN254
  regression as an observed-but-unexplained measurement, not a settled
  result. The 1.75× figure in this report (T.3 / Section 11) remains the
  valid -O2-vs-O2 comparison; the 1.93× is the valid -O3-vs-O3 comparison.

  Energy (added by the -O3 study; this -O2 report did not measure energy):
    AmorE/round 160.17 mJ vs 1× RELIC 85.21 mJ → 1.88× (n=10 replicas).
    PPK2 uncalibrated (no reference resistor); absolute mJ indicative,
    ratio calibration-independent. Source: analysis/results/
    energy_real_20260605.json.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  11.2  Why AmorE still has value

  AmorE is not faster than local C — but speed was never the only
  axis. The protocol's value lies elsewhere:

  Resource                    AmorE client   RELIC pairing   Ratio
  --------------------------- -------------- --------------- -------------
  Flash (code + rodata)       20.0 KB        84.3 KB         4.2x lighter
  SRAM working (data + bss)   3.2 KB         67.3 KB         21x  lighter
  SRAM incl. stack/heap       35.2 KB        99.3 KB         2.8x lighter

  Note (revised 2026-05-31): an earlier version of this table reported
  "33x lighter" on SRAM. That figure compared AmorE's working globals
  (3.1 KB, .data+.bss) against RELIC's *total* SRAM including the 32 KB
  linker stack/heap reservation (101.4 KB) — not a like-for-like basis.
  The table above reports both bases consistently: working globals (the
  memory each implementation actually allocates) give 21x; total SRAM
  including the identical 32 KB stack/heap reservation gives 2.8x. The
  RELIC figures were re-measured on the current easy-C (ARITH=easy)
  build via arm-none-eabi-size -A; the bulk of RELIC's SRAM is its
  pairing context (first_ctx, ~67 KB). Flash is code+rodata+data.

  In addition, AmorE provides:
    - Pairing-library avoidance: client needs no pairing implementation
    - Cheating detection: malicious results rejected with prob ~ 1
    - Feasibility on parts where RELIC does not fit (e.g. STM32L0/G0
      with 4-32 KB SRAM)

  11.3  Honest summary

  The right framing for AmorE on STM32F4 is therefore:

      "Trade ~1.75x speed (vs in-place C) for 21x less working
       RAM, 4.2x less Flash, and the ability to outsource pairing
       computation to an untrusted helper with verifiable results."

  On a Cortex-M4 with 192 KB SRAM, this trade-off is not compelling
  for speed-sensitive applications. It becomes compelling on:
    - smaller MCUs where 100 KB of pairing-state cannot fit
    - settings where memory footprint is critical (no pairing library)
    - settings where battery cost of computation matters more than
      latency

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Two distinct comparisons, each measured on real hardware:

  vs Python server on RPi  (Section 7)
    Speedup            :  192.4x   (systemic, deployment-specific)
    AmorE cost ratio   :  0.52%    of remote pairing time

  vs Optimized C on same STM32  (Section 11)
    Speedup            :  0.57x    (i.e. AmorE is 1.75x SLOWER)
    Memory ratio       :  AmorE uses 21x less working RAM and 4.2x less Flash

    Fair protocol comparison (added 2026-06-01):
    The 1.75x SLOWER figure above compares the AmorE client (hand-written,
    unoptimized Fp12 arithmetic) against RELIC's optimized pairing, so it
    mixes an implementation gap into the protocol result. We measured the
    client's primitives using RELIC's OWN routines on this same STM32
    (microbench validated vs the pairing baseline to 0.008%: 36,775,493 vs
    36,778,389 cyc) and applied the paper's cost formula:
       single (M=1):  -27%   (single-pairing delegation does NOT win on M4)
       batch  M=5:    +25%
       batch  M=10:   +32%
       batch  M=50:   +37%
    Conclusions: ~1.51x of the 1.75x is implementation debt (home Fp12 vs
    RELIC Fp12, closable with RELIC-grade arithmetic); even with RELIC-grade
    arithmetic SINGLE delegation does not win on M4; AmorE wins via BATCH
    delegation (M>=5), +25% to +37% per pairing.

  ──────────────────────────────────────────────────────────────────────
  >>> WHY "-27%" AND "+37%" ARE BOTH CORRECT (not a contradiction) <<<
  -27% (single, M=1): delegating ONE pairing costs 27% MORE -> AmorE loses.
  +37% (batch, M=50): delegating 50 at once SAVES 37%        -> AmorE wins.
  The sign flips because the one-time setup cost is shared across the
  whole batch. One pairing is the worst case; many pairings (batch) is
  the case AmorE is designed for.
  ──────────────────────────────────────────────────────────────────────

  The contribution of AmorE is therefore best characterized as
  ENABLING pairing-based cryptography on memory-constrained MCUs
  where a full pairing library does not fit, with verifiable
  outsourcing of the pairing computation. AmorE is a public-input
  protocol (paper Section 1): pairing arguments are sent in the clear;
  the protocol's security property is verifiability of the server's
  response, not input confidentiality. It is not a speed optimization
  for parts where RELIC already fits comfortably.
